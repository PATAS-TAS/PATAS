# Ответы на вопросы о масштабировании

## 1. Можно ли использовать 10 инстансов для обработки 10M сообщений за 7 часов?

### ❌ Текущее ограничение

**НЕТ, нельзя просто запустить 10 инстансов на одном датасете.**

PATAS использует **distributed locks** для предотвращения одновременной обработки одного и того же набора данных. Lock key: `pattern_mining:{days}:{min_spam_count}` - все инстансы с одинаковыми параметрами будут конкурировать за один lock, и только один сможет работать.

### ✅ Решение: Шардирование данных

**ДА, можно использовать 10 инстансов, если разделить данные на шарды:**

1. **Разделите данные на 10 шардов** (по message_id или timestamp):
   - Инстанс 1: message_id 1-1,000,000
   - Инстанс 2: message_id 1,000,001-2,000,000
   - ...
   - Инстанс 10: message_id 9,000,001-10,000,000

2. **Каждый инстанс использует уникальный lock key:**
   - `pattern_mining:7:10:shard:1`
   - `pattern_mining:7:10:shard:2`
   - ...

3. **Результаты:**
   - Stage 1: ~4 минуты (параллельно) = **~4 минуты общее**
   - Stage 2: ~7 часов (параллельно) = **~7 часов общее**
   - **Total**: ~7 часов (вместо 3 дней на одном инстансе)

**Roadmap:** Автоматическое шардирование в P1 (после успешного пилота). См. [Horizontal Scaling](wiki_Horizontal-Scaling.md) для деталей.

## 2. Минимальные технические требования для 10M сообщений/день

### Вариант 1: Incremental Mining (рекомендуется)

**Обрабатывать только новые сообщения (~1.4M/день):**

- **CPU**: 4-8 vCPU
- **RAM**: 16-32 GB
- **Диск**: 200 GB SSD
- **PostgreSQL**: 32-64 GB RAM, SSD
- **Redis**: 4-8 GB RAM (для distributed locks)
- **GPU**: Опционально (для локальных LLM/embeddings)

**Время обработки:**
- Stage 1: ~6 минут
- Stage 2: ~10 часов
- **Total**: ~10 часов

### Вариант 2: 10 инстансов (параллельная обработка)

**На инстанс:**
- **CPU**: 4-8 vCPU
- **RAM**: 16-32 GB
- **Диск**: 100 GB SSD

**Общая инфраструктура:**
- **PostgreSQL**: 64-128 GB RAM, SSD
- **Redis**: 8-16 GB RAM
- **Load Balancer**: Опционально (для API)

**Время обработки:**
- **Total**: ~7 часов (параллельно)

## 3. Способна ли текущая архитектура к такому масштабированию?

### ✅ Что уже поддерживается:

1. **Distributed locks** - координация между инстансами
2. **Checkpointing** - возобновление обработки
3. **Incremental mining** - обработка только новых сообщений
4. **Параллельная evaluation** - через `shadow_evaluation_parallel_workers`
5. **Фильтрация правил** - через `max_shadow_rules_to_evaluate`

### ❌ Что нужно добавить для шардирования:

1. **Параметры шардирования** в CLI/API:
   - `--shard-id` / `--total-shards`
   - Или `--message-id-range` (min_id, max_id)

2. **Уникальные lock keys** для каждого шарда:
   - `pattern_mining:{days}:{min_spam_count}:shard:{shard_id}`

3. **Объединение результатов** после обработки всех шардов:
   - Дедупликация паттернов
   - Объединение правил
   - Агрегация метрик

**Статус:** Текущая архитектура **поддерживает** горизонтальное масштабирование, но требует **ручного шардирования** данных. Автоматическое шардирование - в roadmap P1.

## 4. Качество паттернов при масштабировании

### Влияние шардирования:

**Потенциальные проблемы:**

1. **Потеря глобального контекста:**
   - Паттерн может быть редким в одном шарде, но частым глобально
   - **Решение:** Минимальный порог `min_spam_count` должен быть достаточно низким

2. **Дублирование паттернов:**
   - Один и тот же паттерн может быть найден в нескольких шардах
   - **Решение:** Дедупликация при объединении результатов

3. **Разные quality tiers:**
   - Паттерн может быть SAFE_AUTO в одном шарде, но REVIEW_ONLY в другом
   - **Решение:** Использовать глобальные метрики при объединении

**Итоговое качество:** Аналогично обработке на одном инстансе, если правильно объединить результаты.

## 5. Минимизация ложных срабатываний

### Многоуровневая защита:

1. **SQL Safety Validation:**
   - Только SELECT запросы (запрещены INSERT, UPDATE, DELETE, DROP)
   - Whitelist таблиц/колонок
   - Проверка на SQL injection
   - Обнаружение "match-everything" правил (coverage >80% = red flag)

2. **LLM Validation** (опционально):
   - Проверка логики правила перед сохранением
   - **Важно:** LLM не является финальной верификацией (слишком дорого)

3. **Shadow Evaluation:**
   - Тестирование правил на исторических данных перед активацией
   - Метрики precision, recall, false positive rate
   - Автоматическая депрециация при деградации

4. **Safety Profiles:**
   - **Conservative**: precision >= 0.95, max 5 false positives
   - **Balanced**: precision >= 0.90
   - **Aggressive**: precision >= 0.85

5. **Quality Tiers:**
   - **SAFE_AUTO**: precision >= 0.98, low FPR
   - **REVIEW_ONLY**: precision >= 0.90 - требует ручной проверки
   - **FEATURE_ONLY**: precision < 0.90

**Реальные метрики:**
- Precision: 0.93-0.97 (conservative profile)
- False positive rate: 0.15%
- Coverage: 5-8% всех спам сообщений

**Рекомендация:** Используйте **conservative profile** для минимизации false positives. Основная защита - это **shadow evaluation** на исторических данных.
